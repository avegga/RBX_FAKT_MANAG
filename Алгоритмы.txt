# 10. Работа с комментариями к графикам
**Алгоритм добавления комментария:**
1. Пользователь кликает на иконку комментариев рядом с нужным графиком.
2. Открывается панель комментариев, отображается история (основной комментарий и ответы).
3. Пользователь вводит текст (до 1000 знаков) и нажимает «Отправить» или просто покидает поле (автосохранение).
4. Новый комментарий сохраняется в БД с указанием автора, времени, chart_id, analysis_type_id.
5. Для всех пользователей обновляется индикатор новых комментариев.

**Алгоритм ответа на комментарий:**
1. Пользователь нажимает «Ответить» под нужным комментарием.
2. Вводит текст ответа, сохраняет (автоматически).
3. Ответ отображается с отступом справа, parent_id указывает на основной комментарий.

**Алгоритм редактирования/удаления:**
1. Кнопки «Редактировать» и «Удалить» доступны только автору комментария.
2. При редактировании изменяется текст и updated_at.
3. При удалении комментарий удаляется из БД, панель обновляется.

**Алгоритм уведомления о новых комментариях:**
1. При появлении нового комментария или ответа у всех пользователей обновляется индикатор на иконке комментариев.
2. Признак is_read сбрасывается для всех, кроме автора.
3. При открытии панели комментариев is_read устанавливается в true для текущего пользователя.
# Алгоритмы для вкладки "Анализ"

В этом файле приведены основные алгоритмы, реализуемые во вкладке "Анализ" приложения. Описания основаны на требованиях из ТЗ, UI и структуры БД.

---

## 1. Загрузка и выбор данных для анализа

**Алгоритм:**
1. Пользователь нажимает "Загрузить файл" или "Использовать загрузку фактов".
2. Данные проходят фильтрацию и предобработку (если источник — "Загрузка фактов").
3. Данные сохраняются в таблицу `analysis_data_tables`.
4. Обновляется состояние интерфейса: отображается таблица данных, активируются элементы управления графиками.
5. Если структура данных несовместима с текущими графиками — выводится предупреждение, несовместимые графики скрываются.

---

## 2. Создание и настройка типа анализа

**Алгоритм:**
1. Пользователь выбирает или создаёт новый тип анализа.
2. В таблице `analysis_types` создаётся новая запись с конфигурацией (JSON).
3. Пользователь добавляет графики (до 4), настраивает их параметры (тип, поля, агрегации, визуализация).
4. Для каждого графика создаётся запись в `analysis_charts`.
5. Все изменения сохраняются по кнопке "Сохранить тип анализа".
6. При удалении типа анализа каскадно удаляются связанные графики и таблицы данных.

---

## 3. Построение и обновление графиков

**Алгоритм:**
1. Для каждого графика из текущего типа анализа выбираются данные из `analysis_data_tables`.
2. Применяются фильтры, сортировка, поиск (по состоянию UI).
3. Выполняется агрегация и группировка по выбранным полям.
4. Формируются данные для визуализации (bar, line, pie, table).
5. График отображается в левой области, напротив — соответствующая таблица данных.
6. При изменении параметров графика, фильтров или данных — график пересчитывается автоматически.

---

## 4. Управление графиками (drag&drop, скрытие, удаление)

**Алгоритм:**
1. Пользователь меняет порядок графиков drag&drop — обновляется поле `order_index` в `analysis_charts`.
2. При скрытии графика (`is_hidden = true`) он не участвует в расчётах, но сохраняется в конфигурации.
3. При удалении графика запись удаляется из `analysis_charts`, UI обновляется.

---

## 5. Сохранение и восстановление пользовательского состояния

**Алгоритм:**
1. При каждом изменении состояния (выбор типа анализа, изменение размеров областей, фильтры, сортировка) обновляется запись в `analysis_user_state`.
2. При запуске приложения состояние восстанавливается из `analysis_user_state` (последний тип анализа, размеры областей, UI-параметры).
3. Источник данных не активируется автоматически — пользователь должен явно выбрать или загрузить данные.

---

## 6. Обработка ошибок и предупреждений

**Алгоритм:**
1. При ошибке загрузки данных, несовместимости структуры, отсутствии данных или других проблемах формируется сообщение.
2. Сообщение выводится через элемент UI `alert_message` поверх соответствующей области.
3. Пользователь может повторить попытку загрузки или изменить параметры.

---

## 7. Печать и экспорт графиков

**Алгоритм:**
1. Пользователь нажимает кнопку "Печать" на карточке нужного графика (`btn_print_chart_{N}`) или кнопку "Печать всех графиков" (`btn_print_all_charts`).
2. Открывается диалоговое окно с выбором формата печати (PDF, PNG, SVG).
3. Пользователь выбирает нужный формат.
4. Пользователь подтверждает действие (кнопка "Сохранить" или "Печать").
5. Формируется файл с изображением выбранного графика или всех графиков в выбранном формате.
6. Пользователь выбирает место сохранения файла.
7. Если график скрыт, кнопка печати для него неактивна.

---

## 8. Миграция и обновление структуры БД

**Алгоритм:**
1. При обновлении приложения проверяется версия структуры БД.
2. Если требуется миграция — выполняются необходимые изменения схемы (добавление/изменение таблиц, полей).
3. Данные и пользовательские настройки сохраняются без потерь.

---

## 9. Локализация и доступность

**Алгоритм:**
1. Все сообщения, подписи и элементы UI отображаются на выбранном языке (ru_RU, en_US).
2. Все действия доступны с клавиатуры, поддерживается масштабирование интерфейса.

---

## Примечания
- Все алгоритмы должны быть реализованы с учётом модульности и изоляции вкладки "Анализ".
- Взаимодействие с другими вкладками — только через явно определённые интерфейсы.
- Для тестирования критичных сценариев использовать автотесты с уникальными id элементов UI.
