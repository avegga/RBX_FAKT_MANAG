# Результаты разработки по спринтам

## Спринт 1. Каркас системы

### Выполненные работы

1. Настроен backend на Python с использованием Flask.
2. Создан каркас frontend-приложения на HTML, CSS и JavaScript.
3. Интегрирована база данных SQLite.
4. Определена структура проекта: UI, API, хранилище и слой обработки.
5. Реализована основная навигация по вкладкам.
6. Добавлены базовые заглушки для разделов:
   - Анализ;
   - Инструкции;
   - Журнал;
   - Настройки;
   - Резерв.
7. Подготовлена схема базы данных и выполнена начальная инициализация.

### Результат

Первый спринт закрыл задачи по созданию базового каркаса приложения. Проект был подготовлен к переходу на рабочую реализацию пользовательских настроек и шаблонов.

## Спринт 3. Загрузка фактов и базовое отображение

### Выполненные работы

1. Реализована рабочая вкладка Загрузка фактов как основная область MVP.
2. Добавлена строка служебных сообщений с уровнями `info`, `warning`, `error`, `success`.
3. Реализована панель элементов:
   - выбор шаблона;
   - отображение текущего файла;
   - загрузка CSV;
   - кнопка сохранения как заглушка следующего спринта;
   - показ и скрытие правой панели;
   - переключатель отображения всех строк.
4. Flask backend расширен API загрузки данных:
   - чтение CSV в UTF-8 with BOM;
   - разделитель `;`;
   - игнор пустых строк до заголовка;
   - переименование дубликатов заголовков по схеме `Имя`, `Имя_2`, `Имя_3`;
   - формирование рабочего набора текущей сессии.
5. Реализовано применение шаблона при загрузке:
   - совпадающим столбцам назначаются типы из шаблона;
   - отсутствующие ожидаемые столбцы фиксируются как частичная загрузка;
   - новые столбцы получают тип `text` по умолчанию.
6. Добавлена базовая валидация значений по типам:
   - `number`;
   - `money`;
   - `date`;
   - `datetime`;
   - `boolean`.
7. Реализована таблица данных с виртуализацией строк и режимом полного отображения по чекбоксу.
8. Реализована нижняя панель Ошибки с выводом предупреждений и ошибок валидации.
9. Правая панель Настройка данных преобразована в контейнер вкладок:
   - Общие;
   - Столбцы;
   - Фильтры;
   - Типы;
   - Обработка.
10. Успешные, частичные и неуспешные загрузки записываются в журнал событий.

### Проверка

1. Проверен запуск приложения через `.venv` Python и ответ `GET /api/bootstrap`.
2. Проверена загрузка тестового CSV с пустой строкой до заголовка и дубликатом столбца `Amount`.
3. Подтверждено автоматическое переименование дубликата в `Amount_2`.
4. Проверена загрузка по шаблону:
   - отсутствующий столбец шаблона переводит загрузку в частичную;
   - новый столбец получает тип `text`;
   - ошибки типов появляются в ответе API и в интерфейсной модели.
5. Подтверждено появление записей `upload_success` и `upload_partial` в журнале.

### Результат

Третий спринт довёл главный пользовательский сценарий до рабочего состояния: теперь пользователь может выбрать шаблон, загрузить CSV, увидеть таблицу, предупреждения, ошибки и базовый контекст настройки данных. Следующий этап разработки — управление типами, фильтрами, видимостью столбцов и логикой обработки данных по полному ТЗ.

## Спринт 4. Типы, ошибки, столбцы, фильтрация

### Выполненные работы

1. Реализована вкладка Общие:
   - скрыть копейки для денежных столбцов;
   - только целые для числовых столбцов;
   - ограничение первичного отображения по количеству строк.
2. Реализована вкладка Столбцы:
   - управление видимостью столбцов;
   - изменение порядка кнопками вверх и вниз;
   - мгновенное применение порядка и видимости к таблице.
3. Реализована вкладка Типы:
   - назначение типа каждому столбцу;
   - пересчёт ошибок по всему рабочему набору при изменении типа;
   - автосохранение типов в выбранный шаблон.
4. Реализована вкладка Фильтры:
   - текстовый фильтр по содержимому;
   - числовой и денежный фильтры по операциям равно, больше, меньше, диапазон;
   - фильтры даты и даты-времени по диапазону дней;
   - логический фильтр Да, Нет, Пусто.
5. Реализована сортировка по заголовку таблицы с переключением `asc`, `desc`, без сортировки.
6. Подсветка ошибочных ячеек переведена на пересчитываемую клиентскую модель данных.
7. Реализована производная модель рабочего набора без потери исходных строк:
   - фильтрация выполняется по полному набору;
   - лимит строк влияет только на отображение;
   - режим Показать все снимает ограничение отображения.
8. Серверная валидация приведена к правилам ТЗ для типов `number`, `money`, `date`, `datetime`, `boolean`.

### Проверка

1. Пройдена статическая проверка `app.py`, `app.js`, `index.html`, `styles.css` без ошибок.
2. Проверен запуск приложения на локальном Flask server после изменений спринта 4.
3. На тестовом CSV подтверждены правила валидации:
   - пустое `number` трактуется как `0` и фиксируется как ошибка;
   - пустое `boolean` трактуется как `Нет` и фиксируется как ошибка;
   - неверные `money`, `date`, `datetime` попадают в ошибки.
4. Подтверждено, что `bootstrap` после загрузки возвращает актуальный набор данных и типизированный шаблон.

### Результат

Четвёртый спринт перевёл рабочую область из режима базового просмотра в режим управления данными: пользователь может менять типы, пересчитывать ошибки, фильтровать, сортировать, скрывать и переставлять столбцы без потери полного рабочего набора. Следующий этап — операции обработки строк и сохранение результата в рамках спринта 5.

## Спринт 5. Обработка и сохранение результата

### Выполненные работы

1. Реализована рабочая вкладка Обработка:
   - режим Парсинг;
   - заглушка Прочее;
   - выбор столбца только из видимых столбцов;
   - ввод разделителя парсинга.
2. Реализован предпросмотр парсинга по текущему отфильтрованному набору:
   - количество удаляемых строк;
   - количество создаваемых строк.
3. Реализовано выполнение парсинга на backend по всем строкам после фильтрации:
   - пустые фрагменты игнорируются;
   - края фрагментов обрезаются;
   - исходные строки заменяются новыми строками по числу непустых фрагментов.
4. Реализован откат последнего парсинга на один шаг в пределах текущей сессии.
5. Реализовано сохранение результата в XLSX:
   - сохраняются только видимые столбцы;
   - сохраняется текущий порядок столбцов;
   - сохраняются только строки после фильтрации;
   - значения выгружаются в отображаемом пользователю виде.
6. Для экспорта добавлена генерация имени файла по шаблону `data_DDMMYY_HHMM.xlsx` с автоматическим суффиксом при совпадении.
7. Реализована обработка ошибок сохранения с сообщением пользователю и записью в журнал.
8. Добавлена фиксируемая зависимость `openpyxl` в `requirements.txt` для воспроизводимой записи XLSX.

### Проверка

1. Backend расширен новыми API для парсинга, отката и сохранения результата, а также поддержкой загрузки CSV и XLSX.
2. Frontend обновлён так, чтобы парсинг использовал текущий фильтрованный набор, а экспорт отправлял в backend уже упорядоченные и отображаемые значения.
3. Проверка должна подтвердить:
   - корректную замену строк после парсинга;
   - доступность одного шага отката;
   - сохранение XLSX в папку результата с корректным именем и набором столбцов.

### Результат

Пятый спринт переводит рабочую область от настройки данных к их изменению и выгрузке: теперь пользователь может разрезать строки по выбранному столбцу, отменить последний шаг обработки и сохранить итоговый набор в XLSX по текущему виду таблицы.

## Спринт 6. Инструкции, журнал, доводка MVP

### Выполненные работы

1. Реализована вкладка Инструкции с чтением одного внешнего текстового файла из пути, заданного в настройках.
2. Добавлена обработка ошибок доступа и формата файла инструкций с отображением сообщения в интерфейсе.
3. Вкладка Журнал доведена до рабочего состояния: для каждой записи показываются дата и время, уровень, тип события и описание.
4. Backend журнала расширен вычислением уровня события и дополнительными записями по предупреждениям и ошибкам типизации.
5. Настройки доведены до немедленной проверки и автоматического сохранения при изменении корректных путей.
6. Подготовлена инструкция локального запуска MVP в [README.md](README.md).

### Проверка

1. Проверка должна подтвердить загрузку и отображение текстового файла инструкций по пути из настроек.
2. Журнал должен показывать загрузки, частичные загрузки, ошибки, сохранения, изменения настроек, ошибки типизации и операции парсинга.
3. Некорректный путь в настройках должен приводить к мгновенной ошибке без сохранения неверного значения.

### Результат

Шестой спринт закрывает обязательный контур MVP: приложение не только загружает, валидирует, фильтрует, обрабатывает и сохраняет данные, но и показывает рабочие инструкции, фиксирует ключевые события в журнале и готово к пользовательской проверке.
