# Техническое задание

## Проект

Система управления фактами

## Версия документа

MVP, однопользовательская версия

## Назначение системы

Система предназначена для локальной однопользовательской работы с табличными фактами: загрузки исходных данных из CSV-файлов, отображения данных в табличной форме, настройки отображения и обработки данных, фиксации ошибок, ведения журнала событий, а также сохранения результата обработки в файл XLSX.

Система должна работать как web-приложение, доступное через браузер на одном персональном компьютере, без полной перезагрузки страницы при пользовательских действиях.

## Ограничения MVP

1. Система работает в однопользовательском режиме.
2. Поддержка загрузки исходных данных реализуется только для CSV-файлов.
3. Поддержка XLS и XLSX при загрузке резервируется для следующих версий.
4. Вкладки Анализ и Резерв реализуются как заглушки с архитектурным заделом под развитие.
5. Защита настроек паролем в MVP не требуется.
6. Загруженный рабочий набор данных хранится только в памяти текущей сессии.

## Описание UI

### 1. Общая структура интерфейса

Интерфейс приложения должен быть русскоязычным. Раскладка интерфейса должна использовать минимальные внешние и внутренние отступы между панелями и границами окна.

Во всех разделах приложения в верхней части окна должна располагаться главная панель. На главной панели слева направо размещаются:

1. Название системы: Система управления фактами.
2. Вкладка Загрузка фактов.
3. Вкладка Анализ.
4. Вкладка Инструкции.
5. Вкладка Журнал.
6. Вкладка Настройки.
7. Вкладка Резерв.

### 2. Вкладка Загрузка фактов

Вкладка Загрузка фактов является основной рабочей областью MVP. Она должна включать:

1. Главную панель.
2. Строку служебных сообщений.
3. Панель элементов.
4. Рабочую область, разделенную вертикальным разделителем на левую панель Таблица данных и правую панель Настройка данных.

Вертикальный разделитель должен позволять изменять ширину панелей. Установленные пользователем размеры должны сохраняться между сессиями.

### 3. Строка служебных сообщений

Строка служебных сообщений располагается под главной панелью.

Требования:

1. Высота строки не более 20 пикселей.
2. Ширина ориентировочно до 50% рабочей области.
3. В строке отображается только одно последнее сообщение.
4. Сообщение сохраняется до следующего действия или системного события.
5. Обновление сообщения выполняется без перезагрузки страницы.

Поддерживаемые уровни сообщений:

1. Информация.
2. Предупреждение.
3. Ошибка.
4. Успех.

Примеры сообщений:

1. Файл не выбран.
2. Выбран файл: <имя файла>.
3. Данные успешно загружены.
4. Таблица очищена.
5. Панель данных скрыта.

### 4. Панель элементов

Панель элементов располагается под строкой служебных сообщений и занимает всю доступную ширину. Элементы размещаются слева направо.

Состав панели элементов:

1. Выпадающий список Шаблон.
2. Информационная строка текущего файла.
3. Кнопка Загрузить.
4. Кнопка Сохранить.
5. Кнопка Показать/Скрыть панель.
6. Чекбокс Отобразить все.

#### 4.1. Элемент Шаблон

Элемент Шаблон представляет собой выпадающий список.

Требования:

1. Значения по умолчанию: Шаблон 1, Шаблон 2.
2. Значение по умолчанию: Шаблон 1.
3. Пользователь может создавать, переименовывать и удалять шаблоны.
4. Удаление шаблона требует подтверждения.
5. Шаблоны хранятся в базе данных SQLite.

Шаблон должен хранить:

1. Имя шаблона.
2. Версию шаблона.
3. Список ожидаемых столбцов.
4. Назначенные типы данных по названиям столбцов.

Если в загружаемом файле отсутствуют ожидаемые столбцы шаблона, система должна:

1. Продолжить работу с имеющимися столбцами.
2. Считать загрузку частичной.
3. Показать предупреждение в строке служебных сообщений.
4. Записать событие в журнал.

Если в файле присутствуют новые столбцы, отсутствующие в шаблоне, система должна:

1. Отобразить их в таблице.
2. Назначить им тип Текстовый по умолчанию.

#### 4.2. Информационная строка текущего файла

Элемент отображает имя текущего выбранного файла.

Правила:

1. До выбора файла отображается текст Файл не выбран.
2. После выбора файла отображается его имя.
3. После перезапуска приложения может отображаться последний успешно загруженный файл, если он существует.

#### 4.3. Кнопка Загрузить

Кнопка открывает диалог выбора файла в папке источника, заданной в настройках, и сразу после выбора запускает загрузку файла.

В MVP поддерживается только формат CSV.

Требования к CSV:

1. Кодировка UTF-8 with BOM.
2. Разделитель ;.
3. Пустые строки до заголовка игнорируются.
4. Первая содержательная строка считается строкой заголовков.
5. Дублирующиеся названия столбцов должны переименовываться по правилу Имя, Имя_2, Имя_3.

После загрузки:

1. Заголовки должны отображаться в таблице.
2. Все столбцы должны становиться доступными в настройках.
3. Должны применяться типы из выбранного шаблона по совпадающим именам столбцов.
4. При частичной загрузке должны применяться правила шаблона.
5. Ошибки и предупреждения должны фиксироваться в журнале и интерфейсе.

#### 4.4. Кнопка Сохранить

Кнопка сохраняет результат обработки в файл формата XLSX.

Правила сохранения:

1. Сохраняются только видимые пользователю столбцы.
2. Сохраняется текущий порядок столбцов.
3. Сохраняются только строки после фильтрации.
4. Значения сохраняются в отображаемом пользователю виде.
5. Для типа Денежный при отображении без копеек сохраняется только целая часть.

Правила размещения файла:

1. Файл сохраняется в папку результата, заданную в настройках.
2. Пользователь не выбирает путь сохранения вручную.
3. Имя файла формируется по шаблону data_DDMMYY_HHMM.xlsx.
4. При совпадении имени добавляется суффикс.

Если файл сохранить невозможно, система должна:

1. Не выполнять сохранение.
2. Показать ошибку в строке служебных сообщений.
3. Записать событие в журнал.

#### 4.5. Кнопка Показать/Скрыть панель

Кнопка управляет видимостью правой панели Настройка данных.

Требование:

1. Состояние скрыта/показана должно сохраняться между сессиями.

#### 4.6. Чекбокс Отобразить все

Чекбокс управляет только объемом отображаемых строк в интерфейсе.

Правила:

1. После загрузки полный файл считывается в рабочий набор текущей сессии.
2. При установленном чекбоксе отображаются все строки рабочего набора после фильтрации.
3. При снятом чекбоксе отображение ограничивается значением поля Кол-во строк на вкладке Общие панели Настройка данных.
4. Состояние чекбокса не сохраняется между сессиями.

### 5. Левая панель Таблица данных

Левая панель должна состоять из двух областей, разделенных горизонтальным разделителем:

1. Верхняя область Данные.
2. Нижняя область Ошибки.

Горизонтальный разделитель должен позволять изменять высоту областей. Размер области Ошибки по умолчанию должен составлять 0,3 от области Данные. Состояние скрыта/открыта и размер области Ошибки должны сохраняться между сессиями.

#### 5.1. Область Данные

Область Данные отображает загруженную таблицу с учетом текущих настроек из правой панели.

На отображение влияют вкладки:

1. Столбцы.
2. Общие.
3. Фильтры.
4. Типы.

Требования:

1. Поддержка вертикальной и горизонтальной прокрутки.
2. Поддержка виртуализации строк.
3. Возможность изменения ширины столбцов пользователем.
4. Сортировка по клику на заголовок столбца.
5. Сортировка влияет на отображение и на порядок строк при сохранении результата.
6. Сортировка не сохраняется между сессиями.
7. Ручное редактирование ячеек пользователем запрещено.
8. Ошибочные ячейки должны визуально выделяться цветом фона.

#### 5.2. Область Ошибки

Область Ошибки отображает ошибки, связанные с данными.

Требования:

1. Если ошибок нет, область остается пустой.
2. Ошибки должны отображаться после загрузки данных.
3. Ошибки должны отображаться после изменения типа столбца.
4. При изменении типа пересчет ошибок должен выполняться по всем строкам файла, а не только по видимым.
5. Каждая ошибка должна содержать номер строки, имя столбца, исходное значение и причину ошибки.

### 6. Правая панель Настройка данных

Правая панель Настройка данных является обязательной частью MVP и должна содержать внутренние вкладки. Панель должна иметь вертикальную прокрутку. Все изменения на вкладках должны применяться мгновенно, без кнопок Применить или Сохранить.

Общие требования:

1. При уменьшении ширины панели элементы не должны накладываться друг на друга.
2. Текст на кнопках должен обрезаться по границам кнопки.
3. При наведении курсора должен отображаться полный текст кнопки.

Состав вкладок:

1. Общие.
2. Столбцы.
3. Фильтры.
4. Типы.
5. Обработка.

#### 6.1. Вкладка Общие

Вкладка содержит общие настройки отображения данных.

Состав:

1. Чекбокс скрыть копейки.
2. Чекбокс только целые.
3. Поле Кол-во строк.

Правила:

1. скрыть копейки действует только для столбцов типа Денежный.
2. только целые действует только для столбцов типа Числовой.
3. Эти настройки независимы друг от друга.
4. Для числового типа при включенном режиме только целые дробная часть отбрасывается.
5. Поле Кол-во строк ограничивает первичное отображение строк при загрузке.

#### 6.2. Вкладка Столбцы

Вкладка управляет видимостью и порядком столбцов.

Состав:

1. Вертикальный список столбцов.
2. Чекбокс слева от каждого столбца.
3. Кнопки вверх и вниз для изменения порядка.

Правила:

1. Установленный чекбокс означает, что столбец отображается.
2. Изменение порядка должно сразу отражаться в таблице.
3. При сохранении должны использоваться только видимые столбцы в текущем порядке.

#### 6.3. Вкладка Фильтры

Вкладка содержит фильтры по каждому столбцу с учетом его назначенного типа.

Поддерживаемые типы для MVP:

1. Текст.
2. Числовой.
3. Денежный.
4. Дата.
5. Дата-время.
6. Логический.

Правила фильтрации по типам:

1. Для типа Текст используется операция содержит.
2. Для типа Числовой используются операции равно, больше, меньше и диапазон от/до.
3. Для типа Денежный используются те же операции, что и для числового типа.
4. Для типа Дата используется фильтрация по диапазону дат без учета времени.
5. Для типа Дата-время фильтрация учитывает только день, без учета времени.
6. Для типа Логический используется список Да, Нет, Пусто.

Правила обработки пустых значений:

1. Для Дата пустое значение трактуется как 01.01.1900 и одновременно фиксируется как ошибка.
2. Для Числовой и Денежный пустое значение трактуется как 0 и одновременно фиксируется как ошибка.
3. Для Логический пустое значение трактуется как Нет и одновременно фиксируется как ошибка.

Общие правила:

1. Фильтры работают по всему рабочему набору, а не только по видимым строкам.
2. Результат фильтрации влияет на текущее отображение.
3. Результат фильтрации влияет на набор строк, попадающих в сохранение.
4. Если столбцу не назначен тип, используется текстовый фильтр.

#### 6.4. Вкладка Типы

Вкладка предназначена для назначения типов данных столбцам.

Состав:

1. Список столбцов.
2. Для каждого столбца выпадающий список выбора типа.

Доступные типы:

1. Текст.
2. Числовой.
3. Денежный.
4. Дата.
5. Дата-время.
6. Логический.

Правила:

1. При изменении типа он применяется сразу.
2. Ошибки должны пересчитываться по всем строкам рабочего набора.
3. Ошибочные значения не удаляются из таблицы.
4. Ошибочные значения остаются видимыми и подсвечиваются.
5. Детали ошибок выводятся в область Ошибки.
6. Если тип для столбца уже задан в выбранном шаблоне, он должен автоматически подставляться при следующей загрузке по этому шаблону.
7. Изменение типа автоматически сохраняется в шаблон.

Правила допустимых значений:

1. Числовой и Денежный принимают значения вида 1234, 1 234, 1234.56, 1 234.56.
2. В рамках MVP десятичный разделитель для Числовой и Денежный только точка.
3. Дата принимает форматы ДД.ММ.ГГГГ и ГГГГ-ММ-ДД.
4. Дата-время принимает форматы ДД.ММ.ГГГГ ЧЧ:ММ, ГГГГ-ММ-ДД ЧЧ:ММ, ГГГГ-ММ-ДДTHH:MM:SS.
5. Логический тип трактует значения true/false, Да/Нет, 1/0, Истина/Ложь.

#### 6.5. Вкладка Обработка

Вкладка предназначена для операций обработки данных. В MVP реализуется режим Парсинг и заглушка Прочее.

Состав:

1. Радиокнопка Парсинг.
2. Радиокнопка Прочее.
3. Для режима Парсинг:
   1. Поле выбора Столбец парсинга.
   2. Поле ввода Разделитель парсинга.
   3. Кнопка Выполнить парсинг.
   4. Область предпросмотра результата.
   5. Кнопка отката последнего парсинга.

Правила режима Парсинг:

1. Пользователь может выбрать только один столбец.
2. Для выбора доступны только столбцы, видимые пользователю.
3. Парсинг применяется ко всем строкам после фильтрации, независимо от лимита отображения.
4. Для каждой строки анализируется значение выбранного столбца.
5. Значение разделяется по заданному разделителю.
6. Пустые фрагменты между подряд идущими разделителями игнорируются.
7. Пробелы по краям фрагментов удаляются.
8. Исходная строка удаляется из текущего рабочего набора и заменяется новыми строками по числу непустых фрагментов.
9. Значения остальных столбцов во всех новых строках копируются из исходной строки.
10. Если после разделения не получено ни одного непустого значения, новая строка не создается.
11. До выполнения операции должен быть доступен предпросмотр.
12. Предпросмотр должен показывать количество удаляемых строк и количество создаваемых строк.
13. Должен поддерживаться откат только последнего парсинга на один шаг назад.
14. Изменения действуют только в пределах текущей сессии.

Режим Прочее:

1. В MVP отображается как пустая заглушка без бизнес-логики.

### 7. Вкладка Анализ

Вкладка Анализ в MVP реализуется как заглушка.

Требования:

1. Должен существовать отдельный раздел интерфейса.
2. Должно отображаться сообщение о том, что раздел находится в разработке.
3. Архитектура должна предусматривать дальнейшее расширение.

### 8. Вкладка Инструкции

Вкладка отображает один внешний текстовый файл.

Требования:

1. Загружается только один файл в формате plain text.
2. Путь к файлу задается в настройках.
3. Содержимое отображается в интерфейсе.
4. При недоступности файла должна выводиться ошибка.

### 9. Вкладка Журнал

Вкладка отображает список событий приложения.

Минимальный состав отображения:

1. Дата и время события.
2. Уровень события.
3. Тип события.
4. Описание события.

В журнале должны отображаться как минимум:

1. Успешные загрузки.
2. Частичные загрузки.
3. Ошибки загрузки.
4. Ошибки сохранения.
5. Изменения настроек.
6. Ошибки типизации.
7. Операции парсинга.
8. Системные предупреждения.

### 10. Вкладка Настройки

Во вкладке Настройки пользователь задает:

1. Путь к папке источника файлов.
2. Путь к папке результата.
3. Путь к внешнему текстовому файлу инструкций.

Требования:

1. Все пути валидируются сразу при вводе или изменении.
2. Пути являются локальными путями к папкам и файлу, а не параметрами подключения к БД.
3. Значения сохраняются между сессиями.
4. Защита вкладки паролем в MVP не требуется.

### 11. Вкладка Резерв

Вкладка Резерв в MVP реализуется как заглушка.

## Описание базы данных

### 1. Общие положения

Для MVP используется SQLite. База данных должна хранить только постоянные сущности, настройки, состояние интерфейса и журнал событий. Загруженные табличные данные не сохраняются в базе как рабочий набор и существуют только в памяти текущей сессии.

### 2. Состав таблиц

В MVP должны использоваться следующие таблицы:

1. templates.
2. template_columns.
3. app_settings.
4. app_state.
5. event_log.

### 3. Таблица templates

Назначение: хранение шаблонов.

Рекомендуемый состав полей:

1. id.
2. name.
3. version.
4. created_at.
5. updated_at.

Требования:

1. Значение поля name должно быть уникальным.
2. version хранит версию шаблона.

### 4. Таблица template_columns

Назначение: хранение ожидаемых столбцов шаблона и их типов.

Рекомендуемый состав полей:

1. id.
2. template_id.
3. column_name.
4. data_type.
5. display_order.
6. is_visible_default.
7. created_at.
8. updated_at.

Связь:

1. Один шаблон templates связан со многими записями template_columns.

Допустимые значения data_type:

1. text.
2. number.
3. money.
4. date.
5. datetime.
6. boolean.

### 5. Таблица app_settings

Назначение: хранение постоянных настроек приложения.

Рекомендуемый состав полей:

1. key.
2. value.
3. updated_at.

В таблице должны храниться как минимум:

1. source_folder_path.
2. result_folder_path.
3. instructions_file_path.

### 6. Таблица app_state

Назначение: хранение состояния интерфейса между сессиями.

Рекомендуемый состав полей:

1. key.
2. value.
3. updated_at.

В таблице должны храниться как минимум:

1. right_panel_visible.
2. right_panel_width.
3. left_panel_width.
4. errors_panel_visible.
5. errors_panel_height.
6. last_template_id.
7. last_success_file_path.

### 7. Таблица event_log

Назначение: хранение журнала событий приложения.

Рекомендуемый состав полей:

1. id.
2. created_at.
3. level.
4. event_type.
5. message.
6. details_json.

Допустимые значения level:

1. info.
2. warning.
3. error.
4. success.

Допустимые категории event_type:

1. file_load.
2. partial_load.
3. file_save.
4. validation_error.
5. parsing.
6. settings_change.
7. template_change.
8. system.

Поле details_json предназначено для хранения дополнительных параметров события, например имени файла, числа ошибок, числа строк после парсинга, причины сбоя и иных служебных сведений.

### 8. Исключения из модели MVP

В MVP не требуются:

1. Таблица пользователей.
2. Таблица сохраненных рабочих наборов.
3. Таблица истории фильтров.
4. Таблица истории сессий.

### 9. Требования к расширяемости

Структура БД должна позволять в будущем добавить многопользовательский режим без полного пересмотра модели. Для этого таблицы настроек, состояния и журнала должны проектироваться так, чтобы в будущем к ним можно было добавить внешний ключ пользователя.

## Описание алгоритмов

### 1. Алгоритм запуска приложения

1. Система загружает настройки из базы данных.
2. Система загружает сохраненное состояние интерфейса.
3. Система открывает вкладку Загрузка фактов.
4. Система восстанавливает размеры панелей и состояние интерфейсных областей.
5. Если ранее был сохранен последний успешно загруженный файл и он существует, система отображает его имя в информационной строке.

### 2. Алгоритм загрузки CSV

1. Пользователь нажимает кнопку Загрузить.
2. Система открывает диалог выбора файла в папке источника.
3. После выбора файла система проверяет формат.
4. В MVP допускается только CSV.
5. Система читает файл с кодировкой UTF-8 with BOM и разделителем ;.
6. Система игнорирует пустые строки до заголовка.
7. Первая содержательная строка интерпретируется как строка заголовков.
8. При наличии дублирующихся названий столбцов система переименовывает их с добавлением суффиксов.
9. Формируется внутренний рабочий набор данных текущей сессии.
10. По совпадающим названиям столбцов применяются типы из выбранного шаблона.
11. Отсутствующие ожидаемые столбцы фиксируются как частичная загрузка.
12. Новые столбцы получают тип Текстовый по умолчанию.
13. Выполняется проверка значений по типам по всем строкам рабочего набора.
14. Результаты отображаются в таблице и в панели ошибок.
15. Система записывает событие в журнал.

### 3. Алгоритм назначения типов столбцов

1. Пользователь изменяет тип столбца на вкладке Типы.
2. Система немедленно применяет новый тип.
3. Новый тип автоматически сохраняется в выбранный шаблон.
4. Система повторно проверяет все значения по всем строкам рабочего набора.
5. Ошибки пересчитываются заново.
6. Ошибочные значения остаются в таблице.
7. Ошибочные ячейки подсвечиваются.
8. Детали ошибок выводятся в область Ошибки.
9. Система записывает событие в журнал.

### 4. Алгоритм проверки значений по типам

1. Текстовый тип принимает любое строковое значение.
2. Числовой тип принимает значения вида 1234, 1 234, 1234.56, 1 234.56.
3. Денежный тип принимает те же форматы, что и числовой.
4. Для числового и денежного типов в MVP используется только точка как десятичный разделитель.
5. Тип Дата принимает форматы ДД.ММ.ГГГГ и ГГГГ-ММ-ДД.
6. Тип Дата-время принимает форматы ДД.ММ.ГГГГ ЧЧ:ММ, ГГГГ-ММ-ДД ЧЧ:ММ, ГГГГ-ММ-ДДTHH:MM:SS.
7. Логический тип принимает значения true/false, Да/Нет, 1/0, Истина/Ложь.
8. Пустые значения для Дата трактуются как 01.01.1900 и одновременно фиксируются как ошибка.
9. Пустые значения для Числовой и Денежный трактуются как 0 и одновременно фиксируются как ошибка.
10. Пустые значения для Логический трактуются как Нет и одновременно фиксируются как ошибка.

### 5. Алгоритм отображения строк

1. После загрузки полный файл помещается в рабочий набор текущей сессии.
2. Если включен чекбокс Отобразить все, отображаются все строки после фильтрации.
3. Если чекбокс выключен, отображается только число строк, заданное в поле Кол-во строк.
4. Ограничение влияет только на интерфейс и не влияет на фильтрацию, пересчет ошибок и парсинг.

### 6. Алгоритм управления столбцами

1. Пользователь включает или выключает видимость столбца на вкладке Столбцы.
2. Таблица обновляется мгновенно.
3. Пользователь изменяет порядок столбцов кнопками вверх и вниз.
4. Новый порядок сразу применяется в таблице.
5. При сохранении используются только видимые столбцы в текущем порядке.

### 7. Алгоритм фильтрации

1. Пользователь задает фильтры по столбцам.
2. Система применяет фильтрацию ко всему рабочему набору.
3. Результат фильтрации используется для отображения таблицы.
4. Результат фильтрации используется при сохранении результата.
5. Для Текстовый используется операция содержит.
6. Для Числовой и Денежный используются операции равно, больше, меньше и диапазон от/до.
7. Для Дата и Дата-время фильтрация выполняется по дню.
8. Для Логический используется выбор Да, Нет или Пусто.

### 8. Алгоритм сортировки

1. Пользователь нажимает на заголовок столбца.
2. Система сортирует строки по выбранному столбцу.
3. Сортировка влияет на текущее отображение.
4. Сортировка влияет на порядок строк при сохранении результата.
5. Сортировка не сохраняется между сессиями.

### 9. Алгоритм парсинга

1. Пользователь выбирает режим Парсинг.
2. Пользователь выбирает один из видимых столбцов.
3. Пользователь задает разделитель.
4. Система анализирует все строки после фильтрации.
5. Для каждой строки значение выбранного столбца разделяется по заданному разделителю.
6. Пустые фрагменты игнорируются.
7. У каждого непустого фрагмента удаляются пробелы слева и справа.
8. Если после обработки получен хотя бы один непустой фрагмент, исходная строка удаляется, а вместо нее создаются новые строки.
9. Значения остальных столбцов копируются из исходной строки в каждую новую строку.
10. До выполнения операции система формирует предпросмотр с числом удаляемых и создаваемых строк.
11. После выполнения операции система сохраняет состояние для отката.
12. Поддерживается только один шаг отката назад.
13. Изменения действуют только в текущей сессии.
14. Система записывает событие в журнал.

### 10. Алгоритм отката парсинга

1. Пользователь нажимает кнопку отката.
2. Система восстанавливает рабочий набор в состояние до последнего парсинга.
3. Система обновляет таблицу, панель ошибок и служебное сообщение.
4. Система записывает событие в журнал.

### 11. Алгоритм сохранения результата

1. Пользователь нажимает кнопку Сохранить.
2. Система берет текущий набор строк после фильтрации.
3. Система применяет текущий порядок столбцов.
4. Система исключает скрытые столбцы.
5. Система подготавливает значения в том виде, как они отображаются пользователю.
6. Для типа Денежный при отображении без копеек сохраняется только целая часть значения.
7. Система формирует имя файла по шаблону data_DDMMYY_HHMM.xlsx.
8. Система проверяет доступность папки результата.
9. Система выполняет сохранение в XLSX.
10. При ошибке записи система прерывает операцию, показывает ошибку и записывает событие в журнал.
11. При успешном сохранении система показывает сообщение об успехе и записывает событие в журнал.

### 12. Алгоритм ведения журнала

1. Каждое значимое пользовательское или системное действие формирует запись в журнале.
2. Запись содержит дату и время, уровень, тип события, сообщение и дополнительные детали.
3. Журнал хранится в SQLite.
4. Пользователь просматривает журнал на отдельной вкладке без возможности редактирования.

### 13. Алгоритм работы настроек

1. Пользователь вводит или изменяет путь в настройках.
2. Система немедленно проверяет существование указанного пути.
3. При корректном пути система сохраняет значение в базе данных.
4. При некорректном пути система показывает ошибку.
5. Изменение настройки записывается в журнал.

### 14. Алгоритм восстановления состояния между сессиями

1. При изменении параметров интерфейса система сохраняет состояние в базе данных.
2. При следующем запуске система читает сохраненное состояние.
3. Система восстанавливает размеры панелей, видимость правой панели и состояние нижней панели Ошибки.

## Рекомендуемые бизнес-кейсы для уточнения реализации

Для дальнейшей детализации реализации рекомендуется отдельно зафиксировать не менее трех бизнес-кейсов:

1. Успешная загрузка CSV без ошибок, настройка отображения и сохранение результата.
2. Частичная загрузка CSV с отсутствующими ожидаемыми столбцами шаблона.
3. Изменение типов, появление ошибок, фильтрация, парсинг и откат последнего парсинга.

Наличие бизнес-кейсов не блокирует старт разработки MVP, но позволит точнее определить содержимое журнала, формулировки пользовательских сообщений, а также ожидаемое поведение в спорных и пограничных ситуациях.
